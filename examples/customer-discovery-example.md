## 1. Discovery Scope

**Product:** Velora — AI-powered code review platform for mid-size engineering teams.

**Discovery type:** Evaluative — testing whether engineering leads at 20-100 person teams will adopt an AI code review tool alongside their existing human review process.

**Mission statement:** Determine whether AI-assisted code review addresses a real bottleneck in mid-size engineering teams' shipping velocity, identify which aspects of code review are most painful, and validate willingness to pay for a tool that reduces review cycle time by 50%+.

**Decision this informs:** Whether to invest Q3 engineering resources in expanding the AI review engine (contextual suggestions, security scanning) or pivoting toward CI/CD integration as the primary value prop.

## 3. Interview Guide (as used)

### Context & Background (3-5 min)

1. Tell me about your team — how many engineers, how many PRs per week?
2. Walk me through what happens from the moment a PR is opened to when it's merged.
3. Who reviews code on your team? Is it assigned, voluntary, or round-robin?

**Follow-up probes:** "How long has it been this way?" / "What changed recently?"

### Problem Exploration (10-15 min)

4. What's the biggest bottleneck in your shipping process right now?
5. Tell me about the last time a PR sat in review for more than 24 hours. What happened?
6. How much time do your senior engineers spend on code review per week? How do they feel about that?
7. What kinds of review comments do they make most often? (style, logic, architecture, security?)
8. Have you tried any automation for code review? What worked, what didn't?
9. When a review catches a real bug vs. a style nit, how does that feel different?
10. What's the cost when a PR takes 3 days to merge instead of 3 hours?

**Follow-up probes:** "Can you give me a specific example?" / "What happened next?" / "How did you decide?"

### Solution Direction (5-8 min)

11. If an AI tool could review code before your human reviewers saw it, what would you want it to catch?
12. What would make you NOT trust an AI code review tool?
13. Would you want AI review to replace a human step, or add a step before human review?
14. At what price point would this be an obvious yes? At what price would you say no?
15. Who else should I talk to who deals with this problem?

## 5. Synthesis Framework

### Tagging Taxonomy

**Pain Points Identified:**
- **P1: Review bottleneck** — PRs waiting >24 hours for first review (8/10 participants)
- **P2: Senior engineer tax** — Most experienced devs spending disproportionate time reviewing, not building (7/10)
- **P3: Context switching** — Reviewing interrupts deep work; engineers batch reviews to end-of-day, creating further delay (6/10)
- **P4: Inconsistent coverage** — Some reviewers check style, others check logic; no consistency (4/10)
- **P5: Onboarding burden** — New hires generate PRs that require extra review time (3/10)

### Affinity Map

**Theme 1: "Review is where velocity goes to die"**
> "We ship fast until the PR is opened. Then it sits. The build takes 4 minutes. The review takes 4 hours." — P3, Engineering Lead, 60-person team

> "I've started telling my team to open smaller PRs just so reviews go faster. That's a workaround, not a solution." — P7, VP Engineering, 45-person team

**Theme 2: "My best engineers are stuck reviewing, not building"**
> "My two staff engineers spend Monday and Thursday mornings just doing reviews. That's 20% of their most productive time on something a tool should handle." — P1, CTO, 80-person team

> "I'd rather have Sarah building the new API than reviewing formatting on a junior's PR. But someone has to do it." — P5, Engineering Manager, 35-person team

**Theme 3: "We'd trust AI if it showed its work"**
> "I don't need AI to be perfect. I need it to explain WHY it flagged something. If I can see the reasoning, I'll trust it for 80% of reviews." — P2, Senior Engineer, 70-person team

> "The linter catches syntax. I want something that catches logic errors and explains them like a senior engineer would." — P9, Tech Lead, 55-person team

**Theme 4: "Price sensitivity is real but not fatal"**
> "$20/seat is steep when I'm already paying for GitHub Enterprise. But if it saves my staff engineers 5 hours a week? That's $500/week of their time. Easy math." — P1, CTO, 80-person team

> "I'd need to see it work for a month before I'd pay anything. Free trial or no deal." — P6, Engineering Lead, 28-person team

### Jobs-to-Be-Done

| Situation | Motivation | Expected Outcome |
|---|---|---|
| When a PR has been open for >4 hours with no review | I want the most common issues caught automatically | So the human reviewer can focus on architecture and logic, and the PR moves faster |
| When I'm reviewing a PR in an unfamiliar part of the codebase | I want context about what this code does and why | So I can give a meaningful review without spending 30 minutes reading surrounding code |
| When a junior engineer opens a PR with common mistakes | I want those mistakes caught before I see the PR | So I spend my review time on teaching, not on pointing out style issues |

## 6. Discovery Report

### Top Validated Pain Points

| Pain Point | Severity (1-5) | Frequency (1-5) | Combined Score |
|---|---|---|---|
| P1: Review bottleneck (PRs waiting >24h) | 4.5 | 4.8 | **21.6** |
| P2: Senior engineer tax (best devs reviewing, not building) | 4.2 | 4.0 | **16.8** |
| P3: Context switching (reviews interrupt deep work) | 3.8 | 4.5 | **17.1** |
| P4: Inconsistent review quality | 2.5 | 3.0 | **7.5** |
| P5: Onboarding review burden | 2.0 | 2.5 | **5.0** |

### Strongest Segment

**Teams of 50-100 developers** have the most acute pain:
- Higher PR volume creates compounding review delays
- Fewer senior engineers per PR (ratio of seniors to juniors is lower)
- More willing to pay ($15-25/seat viable) because the cost of senior engineer time is visible at this scale
- 3 of 5 participants in this segment said they would start a paid trial immediately

### Opportunity-Solution Tree

```
Opportunity: Reduce code review cycle time by 50%+
├── Solution A: AI pre-review (catch style, logic, security before human review)
│   ├── Sub-opp: Explain reasoning (trust driver — validated by H6)
│   └── Sub-opp: Confidence scoring (let teams set automation threshold — validated by H8)
├── Solution B: Review routing (assign PRs to best-fit reviewer automatically)
│   └── Lower priority — only 2/10 participants mentioned this
└── Solution C: Codebase context injection (help reviewers understand unfamiliar code)
    └── Medium priority — 4/10 participants mentioned this as secondary need
```

### Recommended Next Steps

1. **Build:** Invest in AI pre-review with reasoning explanations (Solution A). This addresses the top 2 pain points and the strongest validated hypotheses.
2. **Test further:** Run a `prototype-driven-validation` on the confidence-score threshold UX — H8 is validated but the interaction model is unclear.
3. **Revisit pricing:** H5 (willingness to pay at $20/seat) was inconclusive. Run a `experiment-design` with a Van Westendorp price sensitivity survey among the 50-100 dev segment.
4. **Deprioritize:** Review routing (Solution B) and inconsistency fixes (P4) — low signal.

### Confidence Assessment

**Strength:** 10 interviews across 2 segments with consistent patterns on the top pain points. High confidence in H1, H2, H3.

**Limitations:** All participants were from US-based startups. No enterprise (500+ dev) representation. Pricing signals are directionally useful but sample is too small for confident thresholds.
